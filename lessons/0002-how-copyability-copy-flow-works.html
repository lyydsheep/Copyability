Copyability · Lesson 0002

一次复制，是两个事件接力

学完这一小课，你可以沿着一次 Cmd+C，说清楚脚本如何验证选区、生成纯文本、接管剪贴板，以及失败时为什么不会把页面彻底卡死。

keydown验证 Selection触发 copy写入 text/plain

先建立一个准确心智模型

Copyability 不是在后台持续读取文档，也不是扫描整篇页面。它在页面启动时只注册两个事件监听器，然后等待用户主动按下复制快捷键。

这两个监听器通过三个短期状态变量接力：

状态作用
pendingPlainText把已验证的纯文本从 keydown 临时交给 copy 事件。
copySucceeded记录本次 copy 事件是否真的写入了剪贴板。
failureTimer保证失败提示在 1.5 秒后消失，并避免多个计时器叠加。

第一棒：keydown 判断“这次该不该接管”

监听器先确认按键是没有 Alt/Shift 的 Cmd+C 或 Ctrl+C。然后用 window.getSelection() 取得用户当前选区；浏览器用一个 Selection 表示它，再用一个 Range 表示选区在 DOM 中的连续范围。MDN 的 Selection 文档给出了这两个对象的关系。

选区要通过三层门

端点：开始和结束都必须落在只读 .text-editor 的 .ace-line 内。

块类型：正文、1–3 级标题和普通无序列表可复制；page 与 callout 只作为透明结构容器。

内部节点：TreeWalker 遍历范围内的每个文本节点，防止“起点和终点合法，但中间跨过表格或代码块”。

Range.intersectsNode() 用来判断一个 Feishu 块是否与选区相交；TreeWalker 则只枚举文本节点。二者的行为可分别查阅 Range.intersectsNode() 和 Document.createTreeWalker()。

第二棒：把选区变成一次性纯文本

plainText = selection.toString();
pendingPlainText = plainText;
copySucceeded = false;
document.execCommand("copy");

selection.toString() 只取用户选中的可见文字。文本被放进 pendingPlainText 后，脚本调用 execCommand("copy")，让浏览器进入一次真实的复制流程并触发另一个监听器。

技术债：execCommand() 已被弃用。这里保留它，是因为当前 Chrome 中它能从用户按键处理期间触发带有 clipboardData 的 copy 事件；这条兼容路径必须持续测试，而不是视为永久 API。参见 MDN 的弃用说明。

第三棒：copy 事件写入剪贴板

event.clipboardData.setData("text/plain", pendingPlainText);
event.preventDefault();
event.stopImmediatePropagation();
copySucceeded = true;

浏览器的 copy 事件允许处理器用 clipboardData.setData() 指定剪贴板内容，并用 preventDefault() 取消浏览器原来的复制动作。Copyability 只写 text/plain，所以字体、颜色、图片和链接元数据不会进入结果。MDN 的 copy 事件说明展示了同一种标准写法。

两个事件都使用捕获阶段（true），让脚本在事件沿 DOM 向下传播时尽早处理。只有当 copySucceeded 已经变成 true，外层 keydown 才阻止后续传播，于是飞书的拒绝逻辑没有机会覆盖成功结果。

失败为什么不会变成“按键失灵”

如果选区不受支持，脚本直接返回，把复制完全留给页面。如果选区受支持但提取或写入失败，脚本显示短暂错误提示；因为没有把成功标记设为 true，它不会把外层按键事件当作成功请求吞掉。

无论成功失败，pendingPlainText 都在 finally 中恢复为 null。这是一个重要安全边界：旧文本不能被下一次无关的 copy 事件误用。

先凭记忆回答，再展开核对

问题 1：为什么只检查选区的起点和终点还不够？

因为中间可能跨过表格、代码块等不支持内容，所以还要遍历所有相交块和文本节点。

问题 2：pendingPlainText 为什么必须在 finally 里清空？

因为它只是本次事件接力的一次性数据，清空可防止未来复制误用旧文本。

问题 3：为什么必须等 copySucceeded 后才阻止外层按键事件？

因为失败时应保留页面原行为和重试空间，不能让 Copyability 把复制彻底吞掉。

现在你应该能说出的一句话

Copyability 是一个两事件状态机：keydown 验证并暂存选区纯文本，copy 将它写入剪贴板；只有写入成功后才拦截飞书后续处理。
